# 中文网页智能体平台（V1 精简版）
**用途**：直接交给 Cursor / Claude 编码实现的需求与设计文档  
**版本**：v1.1（精简重构版）  
**日期**：2026-03-22

---

## 0. 这版文档的定位

这不是“最终理想形态”的大而全方案，而是**适合 AI 编程直接落地的 V1 实施文档**。

目标只有两个：

1. **先做出能稳定工作的核心链路**
2. **为后续扩展保留接口，但不把所有复杂度一次性实现**

因此本版会主动删减以下内容到 V2：

- 默认多智能体
- 全量 MCP 接入
- 高风险外部写操作
- 企业级多租户权限系统
- 复杂浏览器自动化编排
- 重型工作流引擎强绑定
- 过早拆分过多微服务

---

## 1. 项目目标

构建一个**中文优先的网页研究型智能体平台**，支持：

1. 对话式下达任务
2. 自动搜索网页并读取正文
3. 结合上传文件进行分析
4. 基于证据生成带引用的中文回答
5. 支持较长任务在后台执行
6. 具备基础可观测、可调试、可评测能力

### 1.1 V1 核心价值

V1 不追求“什么都能自动完成”，只追求把下面 4 类任务做好：

- 单网页 / 多网页资料研究
- 上传文件分析
- 网页 + 文件联合总结
- 带引用的中文报告输出

### 1.2 V1 成功标准

满足以下即视为成功：

- 普通问答首字流出时间 3~8 秒
- 多网页研究任务能稳定完成并给出引用
- PDF / DOCX / Markdown 文件可解析并进入证据池
- 回答中的关键结论可追溯到网页段落或文件分块
- 工具调用成功率明显高于纯 ReAct 裸跑
- 失败时可看到阶段、错误和中间产物

## 4. 核心用户任务

1. “帮我研究某个产品 / 公司 / 技术，给出中文总结和引用”
2. “读取这几个网页，帮我整理差异”
3. “阅读这个 PDF / DOCX，提取重点并总结”
4. “结合网页和我上传的文件，写一份分析报告”
5. “只使用官方来源回答”
6. “这个任务较长，转后台继续跑，完成后展示结果”

---

## 5. 设计原则

1. **证据优先，不先编答案**
2. **单智能体优先，复杂度后置**
3. **程序做确定性工作，模型做判断与生成**
4. **工具按需暴露，不一次性给全套**
5. **所有控制流输出尽量结构化**
6. **先把核心链路做稳，再扩功能**
7. **失败可见，过程可追踪**
8. **引用不是附属功能，而是主链路的一部分**

## 8. 系统模块

## 8.1 Agent Runtime

职责：

- 接收用户任务
- 做意图路由
- 选择模式（fast / deep / background）
- 控制工具循环
- 维护证据池
- 生成最终回答
- 调用校验器

### 运行时状态机

```text
queued
-> routing
-> building_context
-> deciding
-> executing_tool
-> integrating_evidence
-> deciding
-> synthesizing
-> verifying
-> streaming
-> done
```

异常状态：

- tool_error
- timeout
- cancelled
- degraded

### 关键约束

- 单轮最大模型步数：6
- 单轮最大工具调用数：8
- 同一工具连续重复上限：2
- 前台 deep 模式总时限：45 秒
- 后台任务默认时限：15 分钟

---

## 8.2 Intent Router

采用两层：

### 第一层：规则路由
明确特征直接命中：

- 有上传文件 -> file_analysis
- 有 URL -> web_read
- 含“搜索/调研/最新/对比” -> deep_research
- 含“计算/表格/统计” -> calculation
- 超长任务 -> background_candidate

### 第二层：小模型分类
输出固定 JSON：

```json
{
  "primary_intent": "deep_research",
  "requires_web": true,
  "requires_file_parse": false,
  "requires_background": false,
  "tool_bundle": ["web_search", "web_reader", "cite_answer"],
  "risk_level": "low"
}
```

要求：

- 只做分类，不生成答案
- 结构化输出，不能返回自由文本
- 置信度低时允许进入 clarify

---

## 8.3 Context Engine

每次模型调用只注入最小必要上下文：

1. 系统规则
2. 当前任务目标
3. 当前可用工具
4. 最近用户消息
5. 会话摘要
6. 当前证据摘要
7. 输出格式约束

### 裁剪规则

- 最近 3~5 轮消息保留全文
- 更早历史只保留摘要
- 工具原始结果不直接回灌
- 只回灌“摘要 + 元数据 + citation_id”
- 长网页只保留分段摘要，不保留整页全文

---

## 8.4 Model Adapter

统一接口：

```python
class ModelProvider(Protocol):
    async def chat(...)
    async def stream_chat(...)
    async def structured_chat(...)
    async def tool_chat(...)
```

V1 必须支持：

- DeepSeek
- 智谱

适配层负责：

- 统一消息格式
- 统一工具调用格式
- 统一结构化输出
- 统一错误对象
- 统一流式事件

### 模型角色分工

#### 快模型
用于：
- 意图分类
- 查询改写
- 轻量摘要
- 低成本校验

#### 强模型
用于：
- 多步决策
- 证据整合

## 8.5 Tool Runtime

V1 只保留 6 个核心工具：

1. `web_search`
2. `web_reader`
3. `file_parse`
4. `kb_search`
5. `python_exec`
6. `cite_answer`

> 删掉 memory_write、browser_interact、MCP 工具总线、高风险工具。  
> 这些都可以保留接口，但不进入 V1 主链路。

### 工具设计要求

每个工具必须有：

- 明确职责
- JSON Schema 输入
- 结构化输出
- timeout
- retry 策略
- 风险等级
- 日志记录
- 单元测试

### 工具包机制

不把所有工具都暴露给模型，只暴露当前 bundle：

- `search_bundle`: web_search, web_reader, cite_answer
- `file_bundle`: file_parse, kb_search, cite_answer
- `analysis_bundle`: python_exec, cite_answer

## 9. 核心工具定义

## 9.1 web_search

作用：召回候选 URL，不直接作为事实依据。

输入：
- query
- locale
- freshness
- site_allowlist
- top_k

输出：
- results[{title, url, snippet, domain, publish_time}]

规则：
- 同一任务可生成 2~3 条查询
- 搜索结果只用于选页，不直接用于最终回答
- 官方域名优先

---

## 9.2 web_reader

作用：读取网页正文并生成可引用段落。

输入：
- url
- mode: `lite | browser`
- main_content_only
- extract_links

输出：
- title
- final_url
- content_markdown
- paragraphs[{index, text}]
- metadata
- links

规则：
- 先 `lite`，失败再 `browser`
- 去除导航、页脚、广告、重复块
- 为每段生成 locator，如 `p12`

---

## 9.3 file_parse

作用：解析上传文件并形成可检索块。

V1 支持：
- PDF
- DOCX
- XLSX
- PPTX
- TXT
- Markdown

输出：
- plain_text
- sections
- tables
- metadata
- chunks[{chunk_id, text, locator}]

规则：
- PDF 按页码 + 标题切分
- DOCX 按标题层级切分
- XLSX 按 sheet 切分
- PPTX 按页切分

---

## 9.4 kb_search

作用：在已入库文件与历史知识中检索相关块。

输入：
- query
- scope
- top_k

输出：
- hits[{chunk_id, score, text, metadata, citation_id}]

规则：
- 采用 FTS + 向量混合召回
- 重排可先用简单分数融合，V1 不强制单独 reranker

---

## 9.5 python_exec

作用：表格处理、轻量计算、数据清洗。

输入：
- code
- files
- time_limit_sec

输出：
- stdout
- stderr
- artifacts
- tables

限制：
- 容器沙箱
- 默认无外网
- CPU / 内存 / 时长受限

---

## 9.6 cite_answer

作用：基于证据池生成最终答案并绑定引用。

输入：
- evidence_ids
- answer_style
- output_schema

输出：
- summary
- key_points
- risks_or_unknowns
- citations

## 10. 网页研究管道

V1 只做最稳定的研究管道，不做复杂自动交互。

```text
用户问题
-> 路由
-> 生成搜索词
-> 搜索
-> URL 去重与排序
-> 读取候选网页
-> 正文清洗
-> 段落化
-> 证据入池
-> 回答生成
-> 引用校验
-> 输出
```

### 10.1 URL 选择策略
按以下优先级排序：

1. 官方网站 / 官方文档
2. 高质量媒体 / 技术社区
3. 更新时间较新
4. 与问题语义相关
5. 页面可读性高

### 10.2 两段式读取
- `lite`：HTTP + HTML 抽取
- `browser`：Playwright 渲染

只有 `lite` 失败或正文缺失时才切 `browser`。

### 10.3 内容清洗规则
- 去广告 / 导航 / 页脚
- 去重
- 过滤过短段
- 合并断裂段
- 保留段落编号

### 10.4 引用规则
每条关键结论都要能映射到：

- source_uri
- title
- locator
- fetched_at

若证据冲突：
- 不强行下结论
- 输出“来源存在冲突”

## 11. 文件处理与检索

### 11.1 文件流程

```text
上传
-> 保存对象存储
-> 类型识别
-> 文本/表格解析
-> 分块
-> 写入 PostgreSQL
-> 生成 embedding
-> 写入 pgvector
```

### 11.2 V1 分块策略
- 控制 chunk 长度，避免过长
- 保留页码 / 标题路径
- 每个 chunk 都有稳定 locator

### 11.3 混合检索
V1 采用：

- PostgreSQL FTS
- pgvector 相似度
- 简单加权融合

先不要引入额外重排模型，避免链路过重；  
等评测证明必要时再加 reranker。

## 18. 给 Claude 的编码约束

以下要求必须附在任务说明里：

1. **先实现主链路，不同时铺太多边缘功能**
2. **先单智能体，后多智能体**
3. **先实现读网页，不实现复杂网页交互**
4. **所有控制流输出都用 Pydantic Schema 约束**
5. **所有工具都要有单测**
6. **不要在业务代码里写 provider-specific if/else**
7. **不要把工具原始输出直接塞回模型**
8. **所有关键结论都要引用化**
9. **所有失败都要有明确错误对象**
10. **先保证可调试，再做 prompt 花活**

## 20. 最终结论

这套 V1 的核心不是“大而全”，而是：
